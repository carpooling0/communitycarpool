⚡ Edge Functions & Automation

Edge functions reference

All backend logic in Community Carpool runs as Deno-based Supabase Edge Functions. Each function has a specific trigger, JWT verification setting, and purpose. Source files are in Supabse Edge Functions/ (note folder name typo — do not rename).

What verify_jwt means

Each edge function has a verify_jwt setting in its config.toml:

  • verify_jwt: true — Supabase validates the Authorization: Bearer [anon_key] header before the function runs. Requests without a valid JWT are rejected with 401. The anon key must be current and not expired.
  • verify_jwt: false — No JWT check. The function handles its own authentication internally (e.g. checking a token query parameter or deletion token).

Cron jobs that call functions with verify_jwt: true work because Supabase automatically passes a valid service-level JWT when invoking scheduled functions internally.

Functions reference

Function Trigger verify_jwt Description
submit-journey Frontend POST true Main form submission handler. Checks blacklist, validates/updates T&C acceptance, finds or creates user record, creates submission, then calls find-matches internally.
find-matches Called by submit-journey false Spatial matching algorithm. Uses find_nearby_users RPC to find submissions within proximity thresholds. Creates match records. Returns match count. Not called directly by frontend.
get-matches-page Frontend GET ?token= true Returns all matches for a user identified by their journey token. Applies privacy masking — names are partially masked, emails are hidden until mutual_confirmed. Returns 401 if token expired.
update-match-status Frontend POST true Updates match interest status. Handles three transitions: interest_expressed (one side interested), not_interested (decline), mutual_confirmed (both interested — reveals full contact info).
update-email-prefs Frontend GET / POST false GET returns current email preferences for a user (identified by token or email). POST updates preferences. Accepts camelCase body keys: unsubscribedMatches, unsubscribedReminders, unsubscribedMarketing.
manage-deletion Frontend POST false Two-step deletion flow. Step 1: user requests deletion — sends confirmation email with token. Step 2: user confirms via token link — schedules deletion. Unregistered emails silently succeed (no user enumeration).
submit-support-ticket Frontend POST false Creates a support ticket. Rejects ticket type deletion with 400 — users must use the manage-deletion flow. Sends notification to support_notify_email config value.
batch-send-emails Cron (scheduled) true Sends match notification emails to users with unsent matches. Respects testing_mode (skips non-whitelisted users when true) and match_notification_enabled (master off switch). Updates match status to notified.
process-deletions Cron (daily) true Processes scheduled account deletions where the grace period has elapsed (deletion_requested_at + data_retention_days). Deletes user, submissions, matches. Writes to deletion_log.
admin-auth Admin panel false Planned Admin login, logout, and session validation. Handles password verification (PBKDF2), session creation, role/IP/expiry checks.
admin-api Admin panel false Planned Umbrella function for all admin dashboard operations: search users, view tickets, manage blacklist, process deletions, view audit log.
resend-webhook Resend HTTP callback false Planned Handles Resend webhook events (email opened, bounced, clicked). Not yet activated.
confirm-deletion Frontend GET false Handles the deletion confirmation link clicked from email. Validates token and confirms the deletion request.
request-deletion Frontend POST false Initiates a deletion request and sends the confirmation email.

Known bugs fixed (engineering notes)

These bugs were identified and resolved. Documented here so they are not accidentally re-introduced:

PostGIS WKB vs GeoJSON

The Supabase JS client returns GEOGRAPHY columns as WKB hex strings, not GeoJSON. Accessing sub.from_point?.coordinates always returns undefined, causing lat/lng to be 0,0 and no matches to be found. Fix: Use the get_submission_coords(p_id) RPC function which returns float columns (from_lat, from_lng, to_lat, to_lng).

Module-level SES initialisation → 502

Declaring const ses = new SESClient({...}) at the top level of a Deno edge function crashes the function at cold start if AWS secrets are missing, causing a 502 with deployment_id: null. Fix: Wrap in a lazy getter function: function getSES() { return new SESClient({...}); } called only inside the request handler.

!inner + .or() = empty results

Using !inner in a Supabase JS join combined with a .or('sub_a_id.eq.X,sub_b_id.eq.X') filter returns empty results. Fix: Use FK column names directly in the join (e.g. submissions!sub_a_id) and remove !inner.
