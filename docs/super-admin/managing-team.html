Managing team access

The admin panel uses a dedicated role-based access control system separate from Supabase Auth. Admin accounts are stored in the admin_users table and sessions in admin_sessions. Only Super-Admins can add or modify admin accounts.

ℹ️
The admin panel and its authentication system (admin-auth and admin-api edge functions) are planned/in progress. Until deployed, admin operations are performed directly via the Supabase dashboard and SQL Editor.

Admin roles

RoleAccess levelCan do
super_admin Full Everything: manage team, view all data, process deletions, edit config, access audit log, blacklist users, manage organisations
admin Operational View/search users, process deletions, manage support tickets, add to blacklist, view audit log
support Limited View and respond to support tickets only. Cannot access user data or process deletions.

Adding a new admin

Until the admin panel Team section is live, add accounts directly via SQL:

-- Generate a PBKDF2 hash of the password externally, then insert:
INSERT INTO admin_users (email, name, role, password_hash, deletion_pin, is_active)
VALUES (
  'newadmin@example.com',
  'Admin Name',
  'admin',
  '[saltHex:hashHex — generate programmatically]',
  '[hashed 6-digit PIN]',
  true
);

Password hashing uses PBKDF2 via Deno's crypto.subtle: 16-byte random salt, 100,000 iterations, SHA-256, output as hex. Format stored: saltHex:hashHex. Generate this using a small script or the admin-auth edge function's own hashing utility.

Deactivating an admin

-- Immediately prevents login
UPDATE admin_users SET is_active = false WHERE email = 'leavingadmin@example.com';

Sessions belonging to deactivated accounts are rejected on next validation. The account remains in the DB for audit log purposes.

Setting role expiry

For temporary access (e.g. a contractor or seasonal support agent):

UPDATE admin_users
SET role_expires_at = '2026-06-30 23:59:59+04'
WHERE email = 'temp@example.com';

After the expiry date, login will be rejected with "Access expired" even if is_active = true.

IP restriction

To restrict an admin to specific IP addresses:

UPDATE admin_users
SET allowed_ips = ARRAY['203.0.113.10', '198.51.100.0']
WHERE email = 'admin@example.com';

Login attempts from other IPs will be rejected. Leave as NULL (default) to allow login from any IP.

Deletion PIN

A separate 6-digit PIN is required to confirm any "Delete Now" action in the admin panel. This prevents accidental irreversible deletions. The PIN is hashed separately from the login password and stored in admin_users.deletion_pin.

To reset a deletion PIN: update the hashed PIN value in the database using the same hashing method as the password.

Seeding the first Super-Admin

The first Super-Admin account must be seeded directly in the database (there is no admin panel to log into yet). Run the INSERT above with role = 'super_admin'. Once logged in, subsequent accounts can be managed through the Team panel.

Offboarding checklist

When a team member leaves:

  • Set is_active = false on their admin account immediately
  • Rotate any shared secrets they had access to (see Rotating secrets)
  • Remove their SSH keys or personal access tokens from any GitHub account
  • Update shared service logins (Resend, Google Cloud Console, domain registrar) if they used shared credentials
  • Review recent admin_actions log entries for their account
